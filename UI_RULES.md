# Westory UI 규칙

이 문서는 Westory UI 작업 기준이다.
루트 `AGENTS.md`를 먼저 따르고, 이 문서는 시각 구조와 상호작용 원칙에만 추가 적용한다.

## Westory UI/UX 목표
- Westory UI의 목표는 새로워 보이는 것보다 일관되고 즉시 이해되는 것이다.
- 현재 기반은 `Noto Sans KR`, 밝은 회색 배경, 흰 카드, 파란 primary, amber 브랜드 포인트, sticky header다.
- 기존에 잘 동작하는 화면을 한 번에 갈아엎지 않는다.
- 한 번의 작업에서는 한 화면, 한 섹션, 한 패턴씩 점진적으로 정리한다.

## 학생 화면과 교사 화면의 톤 차이
- 학생 화면은 단순하고 직관적이어야 한다. 바로 다음 행동이 보여야 한다.
- 교사 화면은 관리도구형 정돈감을 우선한다. 정보 밀도는 높을 수 있지만 구조는 더 명확해야 한다.
- 톤 차이는 전면 색상 교체보다 정보 구조, 액션 우선순위, 라벨, 간격에서 먼저 만든다.
- 교사 화면을 학생 화면처럼 꾸미지 않는다.
- 학생 화면에 운영자용 용어, 과한 상태 배지, 관리형 다중 패널을 끌고 오지 않는다.

## 공통 레이아웃 원칙
- 페이지 상단에는 현재 화면 목적을 한 줄로 설명하는 제목과 필요한 보조 정보만 둔다.
- sticky header는 유지하되, 추가 sticky 요소가 본문을 가리지 않게 한다.
- 기본 배경은 밝은 회색, 핵심 콘텐츠는 흰 카드 또는 명확한 섹션으로 묶는다.
- 동일 페이지 안에서 max width, gutter, section gap을 흔들지 않는다.
- 학생 화면은 기본적으로 단순한 흐름을 우선하고, 교사 화면만 필요한 경우에만 다단 구성을 쓴다.

## 버튼 규칙
- 한 화면 또는 한 섹션의 대표 행동만 primary로 둔다.
- primary는 파란 계열을 기본으로 유지한다.
- amber는 브랜드 포인트 또는 주의 강조에 제한적으로 쓴다. 기본 CTA 색으로 남발하지 않는다.
- destructive action은 danger tone과 명시적 라벨을 사용한다.
- icon-only 버튼은 `aria-label` 또는 동등한 텍스트 설명이 반드시 필요하다.
- 저장 중, 로딩 중, 비활성 상태는 눈에 띄게 구분한다.

## 카드 규칙
- 카드는 흰 배경, 옅은 border, 얕은 shadow, 명확한 header/body/footer 구조를 기본으로 한다.
- 카드 안에 카드 안에 카드를 반복하지 않는다.
- 단순 수치나 짧은 텍스트까지 모두 카드로 감싸지 않는다.
- 학생 화면은 카드 수를 줄이고, 교사 화면은 카드보다 섹션 구분의 명확성을 우선한다.

## 탭 규칙
- 탭은 같은 데이터 맥락 안에서 보기만 전환할 때 사용한다.
- 활성 탭은 색과 텍스트 둘 다로 분명하게 구분한다.
- 탭 수는 가능하면 2~5개 안에서 유지한다.
- 탭, 모달, 아코디언을 한 데이터에 중첩해 쓰지 않는다.

## 모달 / 패널 규칙
- 짧은 수정, 확인, 생성은 모달을 우선한다.
- 긴 편집, 미리보기, 복합 설정은 별도 패널이나 페이지 섹션을 우선한다.
- 모달 위에 다른 모달을 다시 띄우지 않는다.
- 저장 버튼과 닫기 버튼의 책임을 섞지 않는다.
- 스크롤 주체가 페이지인지 모달인지 명확해야 한다.

## 테이블 / 리스트 규칙
- 교사 화면에서 비교와 운영 처리가 필요할 때만 테이블을 쓴다.
- 학생 화면은 가능하면 리스트, 카드, 단계형 블록을 우선한다.
- 테이블에는 헤더와 정렬 기준이 분명해야 한다.
- 행 클릭과 행 내부 버튼을 동시에 쓸 때는 우선순위를 혼동하지 않게 한다.
- sticky table header는 실제로 긴 목록을 다룰 때만 쓴다.

## 폼 규칙
- label은 placeholder로 대체하지 않는다.
- help text, validation message, 저장 결과 메시지는 input 가까이에 둔다.
- semester, permission, 공개 범위, 삭제 같은 위험 필드는 설명을 더 붙인다.
- 기본값과 현재값을 구분할 수 있게 한다.
- 저장 버튼은 변경 전과 후의 의미가 명확해야 한다.

## 상태 배지 규칙
- 상태 배지는 의미 중심으로 통일한다. 같은 의미에 다른 색과 다른 문구를 계속 만들지 않는다.
- 새 배지 규칙이 두 곳 이상에서 필요하면 공용 매핑 또는 공용 컴포넌트로 옮길 준비를 한다.
- 색만으로 상태를 전달하지 않는다.
- 이모지 배지는 예외적으로만 쓴다. 특히 교사 운영 화면에서는 새 이모지 배지 추가를 억제한다.
- 경고와 성공 색은 실제 상태 의미가 분명할 때만 쓴다.

## 색상 / 타이포 / 간격 규칙
- 기본 글꼴은 `Noto Sans KR`를 유지한다.
- 본문은 400~500, 제목은 700~800 중심으로 단순하게 운용한다.
- 파란 primary와 amber 포인트는 유지하되, 보조 강조색을 계속 늘리지 않는다.
- 페이지 간 상태색 정의가 흩어져 있으면 먼저 정리 방향을 잡고, 페이지별 임시 색을 늘리지 않는다.
- 간격은 4px/8px 리듬을 기준으로 맞추고, 카드 padding과 section gap을 흔들지 않는다.

## 반응형 규칙
- 모바일에서 먼저 깨지지 않아야 한다.
- 학생 화면은 모바일에서 한 열 흐름을 우선한다.
- 교사 화면은 데스크톱에서 다단 구성을 쓸 수 있지만 모바일에서는 순서를 다시 정리한다.
- sticky 요소, 긴 탭, 넓은 테이블은 작은 화면에서 즉시 재배치한다.
- 스크롤 축이 두 개 생기면 구조를 다시 본다.

## 접근성 최소 기준
- 대비가 약한 텍스트와 상태색 조합을 피한다.
- 클릭 가능한 요소는 충분한 크기를 확보한다.
- 키보드로 도달 가능한 흐름을 해치지 않는다.
- icon-only control, close button, upload trigger에는 이름이 필요하다.
- 오류, 경고, 성공 메시지는 색 외의 텍스트로도 전달한다.

## 공용 컴포넌트 재사용 우선
- 같은 패턴이 두 화면 이상에서 반복되면 `src/components/common/*` 재사용을 먼저 검토한다.
- teacher 폴더에서 student 컴포넌트를 더 깊게 끌어다 쓰는 방식은 우선 해법이 아니다.
- 교사와 학생 모두에 필요한 패턴은 공용 레이어로 올린다.
- 새 UI primitive가 필요하면 button, card, tab, modal, table 순으로 작은 단위부터 정리한다.

## 로컬 스타일 남발 금지
- 새 대형 `<style>` 블록 추가는 원칙적으로 금지한다.
- 페이지 로컬 스타일이 반복되면 `src/assets/index.css`, `assets/css/style.css`, 또는 공용 컴포넌트 기준으로 옮긴다.
- 현재 Tailwind CDN과 `@tailwind`가 섞여 있다. 새 styling 체계를 더 추가하지 않는다.
- 스타일 문제를 해결할 때 기존 혼용 상태를 더 키우지 않는다.

## 점진 개선 원칙
- 전면 재디자인보다 현재 화면의 가장 거슬리는 불일치 하나를 줄이는 쪽이 우선이다.
- 잘 동작하는 기능의 DOM 구조와 로직을 UI 통일만을 이유로 크게 뒤집지 않는다.
- 공통화는 반복 패턴이 확인된 뒤에 한다.
- 로그인, 수업자료, 학습지, PDF, 판서, semester, permissions 영역은 시각 수정이라도 보수적으로 진행한다.

## 남용 억제 원칙
- 이모지, 강조색, 카드, 그림자, 배지를 동시에 많이 쓰지 않는다.
- 학생 화면은 특히 시각적 소음을 줄인다.
- 교사 화면은 꾸미기보다 정리된 운영 도구 느낌을 유지한다.
